都是那個怪夢害的！過去的一點一滴又跑了回來，害我沒辦法專心工作，今天又花了不少時間在修 clusters... 是很幹沒錯，但是跟當兵的時候實在無法相提並論。  
  
這是另外一個夢靨，我曾經被長官要求寫一個資料庫，就只因為我看起來會操作電腦。同梯請他的朋友用 [BCB](https://web.archive.org/web/20070710103854/http://www.borland.com/cbuilder/index.html "The ultima developer") 寫了一個，然後最後變成我在弄，我重新用我僅知道的 [SQL](https://web.archive.org/web/20070710103854/http://www.nssn.org/NssnSearch/DisplayRecord.asp?RecordNo=541424 "Standard") 皮毛不但重寫了引擎，也重寫了 UI... 在寫的過程我不曾有任何時間比這時候更想死的。  
  
知道這像什麼嗎？像把不會游泳的人丟入大海，還好終於還是活了過來。離開的時候我發願絕對不回頭，不再回答使用上的問題，因為上級已經指示使用[聯勤系統](https://web.archive.org/web/20070710103854/http://www.mnd.gov.tw/division/~defense/mil/csf/csfinfo/trans.htm "運輸管理")，他們也不該再使用原來的程式。據說當初聯勤運輸資料庫交付下去的架構太大（某種需要大量頻寬的微軟架構, LOL...），導致無法使用（我們當初使用 BCB 真是用對了，套印單據無人能比，同梯的功勞），我的遺毒據說還被多使用了半年，後來網路掛點（因為網路也是我架的，我退伍後無人接手，能活一個月就不錯了），就只能在 [server](https://web.archive.org/web/20070710103854/ftp://ftp.microsoft.com/developr/drg/CIFS/ "SMB server") 上使用，最後當然就掛點了...  
  


```
SQL: Stupid Query Language.

SELECT SUM(MeterialCount1+MeterialCount2+MeterialCount3)
              as SMC,
       SUM(MeterialWeight1+MeterialWeight2+MeterialWeight3)
              as SMW,
       SUM(MeterialVolume1+MeterialVolume2+MeterialVolume3)
              as SMV
FROM trans_m WHERE AllocDate WITHIN Jun AND July

翻譯:
選擇    箱件數量總和當作 SMC 欄位，
        箱件重量總和當作 SMW 欄位，
        箱件體積總和當作 SMV 欄位，
從      空軍運輸資料庫
當      撥發時間介於六月和七月
```
